CI/CD Overview
The philosophy behind shipping code confidently and continuously

Pillar 1
Continuous Integration (CI)
  • Devs merge to a shared branch multiple times per day
  • Each merge triggers automated tests — no human needed
  • Catches integration problems early
Pillar 2
Continuous Delivery (CD)
  • Validated builds are staged for release automatically
  • A human decision triggers the final production push
  • Software is always in a releasable state
Pillar 3
Continuous Deployment
  • Every passing change deploys to production automatically
  • No human approval at the deployment step
  • Requires high test coverage and fast rollback
💡
Key Insight
CI and CD are often discussed together but serve different purposes. CI keeps the team moving fast; CD ensures every result is always safe to ship. Together they eliminate the high-risk "big bang" release.

What Changes Across Environments
Local
  • Docker to mimic production services (Supabase, etc.)
  • Debug logging on, error overlays enabled
  • No secrets — uses placeholder env vars
Dev
  • Shared dev database, real API integrations (sandbox tier)
  • Verbose logging, feature flags may be toggled on
  • Where multiple developers' branches first meet
Staging
  • Production-mirrored DB with anonymised data
  • Same infra config, secrets, and CDN as prod
  • No debug flags — behaves like production
Production
  • Live data, real users, full monitoring active
  • Minimal logging (performance), strict error handling
  • Change here = real consequences
✓
Best Practice
Staging should mirror production exactly — same infrastructure, config, services, and data shape. Any difference is a future production incident waiting to happen.
⚠
Common Mistake
Skipping staging and deploying directly from dev to production removes your last safety net before real users are impacted.

Phase Summaries
Phase 1 — Branch · Commit · Push
  • Developer works in isolation — no shared codebase at risk
  • All iteration happens before touching the team
Phase 2 — PR · Review · Merge · Dev
  • Human layer catches design issues automation can't
  • CI enforces quality automatically — code earns its way to Dev
Phase 3 — Staging
  • Final proof-of-correctness checkpoint before going live
  • Same infrastructure as production — no surprises on release
Phase 4 — Production
  • Code has passed: local CI, human review, Dev integration, and QA
  • Full monitoring active — rollback ready if anything goes wrong
📌
Key Principle
Each handoff in the pipeline is a quality gate. Code doesn't advance unless it passes. This is what makes CI/CD reliable — not speed, but structured verification at every step.

Core Git Concepts
Repositories
  • Local repo = your machine; Remote repo = shared truth
  • Clone remote, work independently, push back
Commits
  • Immutable snapshot with SHA hash, author, timestamp
  • Cannot be altered — permanent auditable record
Pushes
  • Sends local commits to the remote
  • Triggers CI checks and deployment pipelines
Version History
  • git log, git blame, git revert
  • Essential for diagnosing and rolling back issues
Collaboration
  • Parallel branches, no interference
  • Conflicts require manual resolution before merging
Source of Truth
  • Every pipeline reads from the remote repo
  • No code reaches production without passing through Git
💡
Key Insight
Every production incident is traceable to a specific commit. Git's immutable history gives you the power to understand exactly what changed, when, and by whom — and to undo it with precision.

Review Cards
Card A
Reviewer Comments
  • Inline comments on specific lines of the diff
  • Flags logic errors, performance issues, missing edge cases
  • Creates a permanent record of design decisions
Card B
Requested Changes
  • PR is blocked until all issues are addressed
  • Author must re-request review after fixing
Card C
Approval
  • At least one reviewer approves — logged with name and timestamp
  • Critical code may require two or more approvals
Card D
Merge Decision
Merge
Approvals + CI green
Block
CI failing or changes requested
  • Merge commit — preserves branch history
  • Squash — one clean commit
  • Rebase — linear history on main
✓
Best Practice
Keep pull requests small and focused — one feature or fix per PR. Large PRs are harder to review thoroughly, take longer to merge, and hide bugs more easily.
⚠
Common Mistake
Merging before CI finishes to "save time" defeats the purpose of the pipeline entirely and can silently ship broken code to users.

A broken CI pipeline blocks every developer on the team. Treat a failing pipeline like a production outage — fix it immediately, it's never someone else's problem.
⚠
Common Mistake
Skipping tests to ship a feature faster creates technical debt that compounds — future bugs cost far more to diagnose without a test suite to narrow the cause.

Section 7
Deployment Platforms
From merged code to live product — automated, reliable, and always reversible
Staging → Prod
Platform A
Vercel
  • Connects to GitHub / GitLab / Bitbucket
  • Every push creates a Preview Deployment with a unique shareable URL
  • Merging to main auto-deploys to production — zero config
Platform B
Netlify
  • Watches for pushes — every PR gets a Deploy Preview URL
  • Production deploys trigger automatically on merge to main
  • Supports serverless functions and edge deployments
✓
Best Practice
Always have a tested rollback plan before deploying to production. Know exactly how to revert within seconds — don't wait for an incident to figure it out.
📌
Preview Deployments
Preview deployments are a superpower — show your changes to stakeholders before merging, with a real URL and zero production risk.
